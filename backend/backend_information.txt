Deep Saga 3.0 Backend Information
==================================

This backend is a Node.js and Express API for the Deep Saga 3.0 game.
It handles user accounts, login, player profile data, player stats, player persona,
skill tracking, skill unlocking, database access, and AI prompt setup.


1. Main Technology Used
-----------------------

- Node.js: JavaScript runtime used to run the backend.
- Express.js: Web server framework used to create API routes.
- MySQL / MariaDB: Database used to store users, players, stats, and skills.
- mysql2: Node package used to connect to MySQL.
- bcryptjs: Used to hash user passwords before saving them.
- jsonwebtoken: Used to create and verify login tokens.
- cors: Allows frontend apps to call this backend.
- dotenv: Loads settings from backend/.env.
- @google/generative-ai: Used for Gemini AI narration/prompt features.
- nodemon: Development tool that restarts the server when files change.


2. How The Backend Starts
-------------------------

The server starts from:

  server.js

Startup flow:

1. server.js loads environment variables from .env.
2. server.js imports the Express app from app.js.
3. server.js imports the MySQL pool from config/db.js.
4. Before listening for requests, it tests the MySQL connection.
5. If MySQL connects successfully, the app starts on the PORT from .env or 5000.
6. If MySQL fails, the backend logs the startup error and stops.

Default local URL:

  http://localhost:5000


3. Express App Setup
--------------------

The main Express app is in:

  app.js

app.js does these things:

- Creates the Express application.
- Enables CORS with app.use(cors()).
- Enables JSON request bodies with app.use(express.json()).
- Adds a root health check route:

  GET /

  Response:
  { "message": "Deep Saga 3.0 backend is running" }

- Mounts authentication routes at:

  /api/auth

- Mounts player routes at:

  /api/player


4. Environment File
-------------------

The backend uses:

  .env

Expected environment variables:

  DB_HOST
  DB_USER
  DB_PASS
  DB_NAME
  JWT_SECRET
  GEMINI_API_KEY
  PORT

Important:

- DB_HOST, DB_USER, DB_PASS, and DB_NAME are used by config/db.js.
- JWT_SECRET is used to sign and verify login tokens.
- GEMINI_API_KEY is used by config/gemini.js.
- PORT is optional. If missing, the server uses 5000.
- Do not commit real .env secrets to a public repository.


5. Database Connection
----------------------

The database connection is configured in:

  config/db.js

It creates a MySQL connection pool with:

- host from DB_HOST
- user from DB_USER
- password from DB_PASS
- database from DB_NAME
- connectionLimit of 10

Other files import this pool when they need to run SQL queries.


6. Authentication System
------------------------

Authentication routes are in:

  routes/auth.js

Available routes:

  POST /api/auth/register
  POST /api/auth/login


Register Route
--------------

Endpoint:

  POST /api/auth/register

Expected body:

  {
    "username": "playername",
    "email": "player@example.com",
    "password": "password123",
    "name": "Character Name"
  }

What it does:

1. Checks that username, email, and password exist.
2. Opens a database transaction.
3. Checks if the username is already used.
4. Checks if the email is already used.
5. Hashes the password with bcrypt.
6. Inserts a new row into users.
7. Creates a matching row in players.
8. Uses starting player values from config/gameConfig.js.
9. Creates base condition stats for the new player.
10. Commits the transaction.
11. Returns a JWT token and user data.

The token lasts for 7 days.


Login Route
-----------

Endpoint:

  POST /api/auth/login

Expected body can use identifier, email, or username:

  {
    "identifier": "playername",
    "password": "password123"
  }

What it does:

1. Finds the user by email or username.
2. Compares the submitted password with the saved password hash.
3. If valid, creates a JWT token.
4. Returns the token and user data.


7. JWT Middleware
-----------------

The auth middleware is in:

  middleware/authMiddleware.js

Protected player routes use this middleware.

It expects this HTTP header:

  Authorization: Bearer YOUR_TOKEN_HERE

What it does:

1. Reads the Bearer token from the Authorization header.
2. Verifies the token using JWT_SECRET.
3. Stores the decoded user data in req.user.
4. Allows the request to continue.

If the token is missing, it returns 401.
If the token is invalid or expired, it returns 403.


8. Player Routes
----------------

Player routes are in:

  routes/player.js

All player routes require a valid login token.

Available routes:

  GET  /api/player/profile
  GET  /api/player/skills
  POST /api/player/allocate-stats
  POST /api/player/persona


Player Profile
--------------

Endpoint:

  GET /api/player/profile

What it returns:

- Player id
- User id
- Persona
- Character name
- Race
- Title
- Level
- Experience
- Stat points
- HP and max HP
- Strength
- Dexterity
- Stamina
- Intelligence
- Charisma
- Wisdom
- Current floor
- Current area
- Life number
- Alive state
- Survival year, day, and hour
- Username and email
- time_of_day value calculated from current_hour

time_of_day comes from services/timeEngine.js.


Player Skills
-------------

Endpoint:

  GET /api/player/skills

What it does:

1. Finds the player for the logged-in user.
2. Ensures the player has the base condition stat rows.
3. Gets unlocked player skills.
4. Gets condition stat progress.
5. Returns both skills and condition_stats.


Allocate Stats
--------------

Endpoint:

  POST /api/player/allocate-stats

Expected body:

  {
    "strength": 1,
    "dexterity": 0,
    "stamina": 2,
    "intelligence": 0,
    "charisma": 0,
    "wisdom": 0
  }

What it does:

1. Converts each stat value into a number.
2. Rejects negative values.
3. Rejects non-whole numbers.
4. Requires at least one point to be spent.
5. Opens a database transaction.
6. Loads the current player stats.
7. Checks that the player has enough stat_points.
8. Adds the requested points to the correct stats.
9. Reduces stat_points by the amount spent.
10. Recalculates max HP with:

    max_hp = 20 + stamina * 5

11. Keeps current HP from going above max HP.
12. Saves the updated player.
13. Returns the new stat values.


Update Persona
--------------

Endpoint:

  POST /api/player/persona

Expected body:

  {
    "persona": "ADMIN"
  }

Allowed personas:

  ADMIN
  TRICKSTER
  SENSEI

What it does:

1. Validates the persona.
2. Updates the player's persona in the players table.
3. Returns the selected persona.

10. Time Engine
---------------

Time logic is in:

  services/timeEngine.js

It contains:

- getActionTimeCost(actionKey)
- applyTime({ year, day, hour, hoursToAdd })
- getTimeOfDay(hour)

Action time costs:

- move: 1 hour
- look: 1 hour
- rest: 2 hours
- hide: 1 hour
- appraise: 0 hours
- attack: 1 hour
- defend: 1 hour
- typed: 1 hour

getTimeOfDay returns:

- morning for hours 5 through 11
- afternoon for hours 12 through 16
- evening for hours 17 through 20
- night for every other hour


11. Skill Engine
----------------

Skill logic is in:

  services/skillEngine.js

It handles:

- Creating base condition stats for each player.
- Tracking behavior such as hiding, resting, appraising, typed actions, enemy kills, and high-risk actions.
- Creating dynamic skills when repeated behavior reaches thresholds.
- Unlocking matching skills based on player level or condition stats.
- Returning player skills.
- Returning player condition stat progress.


Base Condition Stats
--------------------

New players get these base condition stats:

- enemies_killed
- times_hidden
- times_rested
- times_appraised
- typed_actions_count
- intent:precision_attack
- intent:environment_attack
- intent:hide
- intent:escape
- intent:devour_remains
- intent:social
- risk:high


Dynamic Skill Examples
----------------------

The backend can create skills from repeated patterns.

Examples:

- Defeating the same enemy enough times can create an enemy slayer skill.
- Defeating the same enemy type enough times can create a hunter skill.
- Hiding enough times can create Shadow Instinct.
- Creative typed actions can create Adaptive Insight.
- Precision attacks can create Weakpoint Reader.
- Environment attacks can create Room Weaponizer.
- Devouring remains can create Predator Metabolism.
- Social attempts can create Creature Reader.
- High-risk actions can create Berserker Commitment.

Dynamic thresholds are stored in services/skillEngine.js.

  config/gemini.js

Current configured model:

  gemini-3.1-flash-lite-preview

This config exports:

- model
- genAI


13. AI Prompt Config
--------------------

AI narrator persona prompt logic is in:

  config/prompts.js

It defines three narrator personas:

- ADMIN: cold, analytical, system-style narrator
- TRICKSTER: playful, mocking, dangerous narrator
- SENSEI: stern, martial mentor narrator

The buildPrompt function creates a strict JSON-only prompt for AI narration.

Expected AI output format:

  {
    "narration": "string",
    "choices": ["string", "string", "string", "string"]
  }

The prompt tells the AI:

- It is only the narrator and choice suggester.
- It must not change game logic.
- It must not invent or modify stats.
- It must use only provided backend context.
- It must return valid JSON only.
- It can return up to 5 choices.


14. Database Files
------------------

The backend schema is maintained through ordered migration files:

  migrations/001_initial_schema.sql
  migrations/002_monster_reincarnation_rpg.sql
  migrations/003_single_rpg_flow_and_10_floors.sql

migrations/001_initial_schema.sql:

- Clean migration file for creating the schema.
- Uses CREATE TABLE IF NOT EXISTS.
- Good for setting up a new database.

migrations/002_monster_reincarnation_rpg.sql:

- Adds monster reincarnation save data.
- Adds action history and an expandable RPG content catalog.

migrations/003_single_rpg_flow_and_10_floors.sql:

- Adds current_floor to the RPG save.
- Seeds the 10-floor descent content.


15. Database Tables
-------------------

users
-----

Stores account login information.

Important columns:

- id
- username
- email
- password
- role
- created_at
- updated_at

username and email are unique.
password stores the bcrypt hash, not the plain password.


players
-------

Stores the game character connected to a user.

Important columns:

- id
- user_id
- persona
- name
- current_race
- current_title
- level
- exp
- stat_points
- hp
- max_hp
- strength_stat
- dexterity_stat
- stamina_stat
- intelligence_stat
- charisma_stat
- wisdom_stat
- current_floor
- current_area
- life_number
- is_alive
- year_survived
- day_survived
- current_hour
- created_at
- updated_at

players.user_id links to users.id.
If a user is deleted, their player is deleted too.


player_condition_stats
----------------------

Stores player behavior counters used for condition-based skill unlocking.

Important columns:

- id
- player_id
- stat_key
- stat_value
- metadata_json
- created_at
- updated_at

Each player can only have one row per stat_key.


skills
------

Stores available skills.

Important columns:

- id
- skill_key
- name
- description
- skill_type
- unlock_type
- required_level
- condition_stat_key
- condition_threshold
- effect_json
- is_dynamic
- source_pattern
- created_at
- updated_at

skill_type can be:

- passive
- active

unlock_type can be:

- level
- condition


player_skills
-------------

Links players to unlocked skills.

Important columns:

- id
- player_id
- skill_id
- is_active
- unlocked_at

Each player can only unlock the same skill once.

17. How To Run
--------------

From the backend folder:

  npm install

Or for normal start:

  npm start

The backend expects MySQL/MariaDB to be running and the database tables to already exist.


18. Basic Request Flow
----------------------

Register flow:

1. Frontend sends POST /api/auth/register.
2. Backend validates username, email, and password.
3. Backend hashes password.
4. Backend creates a user.
5. Backend creates a player.
6. Backend creates condition stat rows.
7. Backend returns JWT token.

Login flow:

1. Frontend sends POST /api/auth/login.
2. Backend checks username/email and password.
3. Backend returns JWT token.

Protected player flow:

1. Frontend sends a request with Authorization: Bearer token.
2. authMiddleware verifies the token.
3. routes/player.js uses req.user.userId to find the player.
4. Backend returns or updates player data.


19. Current Backend Scope
-------------------------

This backend currently covers:

- Account registration
- Login
- JWT authentication
- Player profile retrieval
- Skill retrieval
- Condition stat retrieval
- Stat point allocation
- Persona selection
- Starting player setup
- Database schema setup
- Gemini prompt configuration
- Skill progression helper functions
- Time helper functions

Some game systems appear prepared in services but are not exposed as API routes yet.
For example, skillEngine.js has behavior tracking and dynamic skill progression helpers,
but the visible routes currently only expose profile, skills, stat allocation, and persona.
